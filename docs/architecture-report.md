# گزارش معماری و روش تصمیم‌گیری — دستیار هوشمند بانکی

## ۱. روش استخراج قواعد از بخشنامه‌ها

بخشنامه‌های بانکی به صورت ساختاریافته (JSON) مدل‌سازی شده‌اند. هر بخشنامه شامل مجموعه‌ای از **شرایط** است که بر اساس ویژگی‌های مشتری (سن، جنسیت، شغل، درآمد، سطح ریسک، سابقه پرداخت و گردش حساب) تعریف شده‌اند.

### ساختار قواعد

هر قاعده شامل سه بخش است:
- **فیلد (Field):** ویژگی مشتری مانند `age`، `monthly_income`، `risk_level`
- **عملگر (Operator):** `gte` / `lte` / `eq` / `in` / `not_in` و …
- **مقدار آستانه (Value):** مقدار مرجع برای مقایسه

### نگاشت محصول ↔ شرایط اهلیت

هر محصول بانکی (۱۰ محصول) حداقل یک بخشنامه دارد که شرایط اهلیت آن را مشخص می‌کند:

| محصول | شرایط کلیدی |
|-------|------------|
| وام شخصی | سن ۱۸-۶۵، ریسک کم/متوسط، درآمد ≥ ۱۰M، حداکثر ۱ قسط معوق |
| وام مسکن | سن ۲۱-۶۰، فقط ریسک کم، درآمد ≥ ۳۰M، گردش ≥ ۵۰۰M، بدون معوقه |
| دسته‌چک | سن ۱۸+، ریسک کم/متوسط، گردش ۳ماهه ≥ ۱۰۰M، بدون معوقه، شاغل |
| کارت اعتباری | سن ۱۸-۷۰، ریسک کم/متوسط، درآمد ≥ ۱۵M، حداکثر ۲ معوقه |
| وام کسب‌وکار | سن ۲۱-۶۰، شغل آزاد/مدیر، ریسک کم، درآمد ≥ ۵۰M |
| وام ازدواج | سن ۱۸-۴۰، ریسک کم/متوسط، بدون معوقه |
| سپرده‌ها | سن ۱۸+ (محدودیت کم) |
| خدمات پایه | سن ۱۵+ |

جزئیات کامل: [product-eligibility-mapping.md](product-eligibility-mapping.md)

## ۲. معماری دستیار

سیستم از سه لایه تشکیل شده:

### لایه ۱: بک‌اند Go
- **endpoint محلی RBCI:** contract هویتی، مالی و ریسک با PostgreSQL در Docker
- **موتور قواعد (Rule Engine):** ارزیابی قطعی (deterministic) شرایط اهلیت — بدون LLM
- **تحلیل شکاف:** برای هر شرط ناموفق، فاصله بین مقدار فعلی و مقدار لازم

**دلیل انتخاب Go:** همزمانی بالا، سرعت پردازش، مصرف کم حافظه

### لایه داده محلی RBCI
- در استقرار Docker، PostgreSQL منبع حقیقت مشتریان است.
- `GET/POST/PUT/DELETE /api/rbci/customers` برای مدیریت ورودی‌های محلی RBCI استفاده می‌شود.
- مسیرهای تطبیق (`/api/identity`, `/api/financial`, `/api/rbci`, `/api/match`) از همین store می‌خوانند.
- seedهای PDF فقط یک بار در volume خالی درج می‌شوند؛ حذف یا ویرایش آن‌ها پایدار می‌ماند.
- هیچ sync دوطرفه با RBCI آنلاین در نسخه دمو وجود ندارد. هنگام اتصال RBCI آنلاین، adapter همین contract را پیاده‌سازی می‌کند.
- این تصمیم عمدی است: UI، agent و rule engine فقط با contract محلی RBCI کار می‌کنند، پس تعویض PostgreSQL با RBCI آنلاین نباید منطق اهلیت یا API مصرفی UI را عوض کند.

### مسیر اتصال به RBCI آنلاین
1. در `backend/internal/data` یک adapter HTTP برای RBCI آنلاین جایگزین adapter PostgreSQL کنید.
2. امضاهای فعلی `GetIdentity`، `GetFinancial`، `GetRisk`، `ListCustomers`، `CreateCustomer`، `UpdateCustomer` و `DeleteCustomer` را حفظ کنید.
3. اگر RBCI آنلاین write API دارد، create/update/delete را به همان API push کنید. اگر read-only است، این مسیرها باید `405 Method Not Allowed` یا `501 Not Implemented` برگردانند.
4. mapping فیلدهای RBCI آنلاین را به مدل‌های داخلی `IdentityProfile`، `FinancialProfile` و `RiskAssessment` محدود نگه دارید.
5. تست‌های matching و سناریوهای PDF باید بدون تغییر rule engine پاس شوند؛ فقط تست adapter عوض می‌شود.

### لایه ۲: ایجنت Python (LangChain / LangGraph)
- **۸ ابزار (Tool):** اتصال به هر سرویس بک‌اند
- **حافظه جلسه:** `InMemorySaver` + `thread_id`
- **مدیریت جریان:** تشخیص مشتری موجود/غیرموجود، هدایت مکالمه
- **توضیح‌پذیری:** هر پاسخ شامل ارجاع به شماره بخشنامه

**مدل زبانی:** OpenAI-compatible — سازگار با Ollama (محلی)، vLLM، OpenAI  
**پروتکل:** Chat Completions (`USE_RESPONSES_API=false`)

### لایه ۳: رابط کاربری React + Gateway
- **SPA:** `web/` (TypeScript) — سرو از `agent/server.py` (FastAPI)
- **دو حالت:** بررسی سریع (مستقیم به موتور) + دستیار هوشمند (LLM)
- **RTL فارسی** و نمایش ساختاریافته: مجاز / غیرمجاز / شکاف / تعهدات / افر مشروط

```
React SPA ──► FastAPI gateway ──► Go backend :8080 ──► PostgreSQL local RBCI
                 │
                 └──► LLM (Chat Completions)
```

## ۳. منطق تصمیم‌گیری

### الگوریتم تطبیق
1. دریافت پروفایل مشتری (هویتی + مالی + ریسک)
2. تبدیل به نقشه فیلد-مقدار (`BuildProfile`)
3. برای هر محصول: ارزیابی تمام شرایط بخشنامه
4. همه شرایط برقرار → **مجاز** + امتیاز تناسب + تعهدات/سقف
5. شرط ناموفق → **غیرمجاز** + Gap + advice + alternatives
6. مرتب‌سازی محصولات مجاز (با boost هدف مراجعه) → افر شخصی‌سازی‌شده

### ارزیابی ریسک Cold-Start (غیرمشتری)
- شروع از امتیاز ۵۰ (ریسک متوسط)
- تعدیل بر اساس سن، شغل، نوع اشتغال، درآمد تقریبی
- محدوده نهایی: ۵ تا ۹۵
- ≤ ۳۵ → کم | ۳۶-۶۴ → متوسط | ≥ ۶۵ → بالا
- خروجی اهلیت با `is_conditional` و `conditions_fa`

### پیامدهای عدم پرداخت
- **کم → متوسط:** تعلیق دسته‌چک، نیاز به ضمانت اضافی
- **متوسط → بالا:** رد تمام وام‌ها، مسدودی دسته‌چک، اقدام حقوقی
- **بالا:** محدودیت کامل، گزارش به سامانه اعتبارسنجی

## ۴. مدیریت خطا و اعتبارسنجی

- کد ملی نامعتبر → `400`
- مشتری یافت‌نشد → `404` (مسیر cold-start)
- قطع سرویس مالی/RBCI → `503` + `upstream_errors`
- ایجنت: timeout/connect → پیام فارسی بدون ادامه با داده ناقص

## ۵. انطباق با الزامات PDF

- ماتریس محصول↔اهلیت و Matching Engine
- Gap Analysis + alternatives + obligations + credit_limit
- Cold-start RBCI و افر مشروط
- سناریوهای خانه‌دار / مدیر / کارمند / غیرمشتری / عدم‌پرداخت
- Agent با Tools، Memory، explainability با `circular_refs`
- UI کارمند شعبه + Ollama/API
- مستندات: OpenAPI، Postman، گزارش معماری، اسکریپت دمو

## ۶. استقرار

```bash
docker compose up -d --build
```

Compose دو سرویس می‌سازد: `app` برای بک‌اند Go + gateway و `postgres` برای endpoint محلی RBCI. سرویس عمومی روی `http://localhost:18080` است.
